# Legacy Staff Evaluation Admin System 中文业务摘要

## 项目状态
- Buildability Score: 72 / 100
- Status: Review Required

## 这个旧系统主要包含哪些模块？

目前资料可以识别出五个模块：Dashboard、员工管理、评估记录、报告复核、权限设置。其中员工管理、评估记录和权限设置证据较充分；报告复核主要来自导航和业务说明，仍需要补充页面或规则确认。

## 管理哪些核心数据？

系统核心数据包括员工、部门、小组、评估记录、角色、权限、账号和操作日志。员工和评估记录来自 mock 页面与 CSV，置信度较高；账号和操作日志主要来自角色说明、提交/审核字段和业务说明，需要继续确认。

## 有哪些角色和权限？

已识别六类角色：总裁、公司管理员、部门主管、组长、评估专员、只读账号。权限设计需要同时考虑操作权限和数据范围。当前较大的不确定点是：部门主管是否能跨部门查看、谁可以导出、谁可以最终审批、组长是否能编辑已提交记录。

## 流程大概如何运转？

评估记录流程大致为：draft -> submitted -> reviewed -> approved，也可能进入 rejected。草稿通常由组长或评估专员创建，提交后由部门主管复核。最终 approval 的角色、退回后是否回到草稿、提交后是否还能编辑，都需要业务方确认。

## 哪些规则现在不确定？

高优先级问题包括部门主管跨部门访问、组长编辑已提交记录、最终审批角色、导出权限和是否必须 approval 才能成为最终记录。中优先级问题包括退回记录流转、操作日志可见性、离职或停用员工是否进入报表、员工是否只停用不删除。

## 目前是否适合进入开发？

当前资料已经可以形成初版开发规格，但权限范围、导出权限、最终审批规则和操作日志可见性仍需业务方确认。因此当前状态为 Review Required。建议先完成业务确认，再把最终 handoff pack 交给 AI coding agent 或开发者实现。

## 下一步业务方需要确认什么？

优先确认角色数据范围、导出权限、最终审批角色、提交后编辑规则和操作日志可见性。确认后可重新生成最终规格包，并将验收标准转成开发测试用例。
