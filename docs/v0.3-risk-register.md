# v0.3 Risk Register

Status: Phase 27A risk register with CI completion note.

This register covers the proposed v0.3 baseline and stretch areas. It does not start implementation.

Phase 26 planning outcome:

- CI implementation requires separate approval.
- Review/apply polish must not become auto-apply.
- Handoff polish must not imply implementation authorization.
- Workflow inference must preserve uncertainty.

Phase 27A outcome:

- CI implementation was separately approved.
- CI remains validation-only.
- Future CI expansion still requires separate approval.

| Risk | Impact | Mitigation | Review Point |
|---|---|---|---|
| CI scope creep into release automation | CI work could accidentally create tag, release, publish, or deployment behavior | Keep CI focused on validation only; Phase 27A adds no publish, release, tag, deploy, secrets, artifact, or schedule behavior | Before any CI expansion |
| Review UX polish becoming auto-apply behavior | Users may think clearer review/apply-plan output authorizes automatic changes | Keep review outputs human-readable and manual; keep auto-apply out of baseline | Before Phase 27B |
| Handoff readability being mistaken as implementation authorization | Handoff packs could be treated as instructions to start coding | Preserve context-only language and no-agent/no-code-generation boundaries | Before Phase 27C |
| Workflow inference overclaiming uncertain transitions | Heuristics could turn weak signals into false workflow requirements | Keep uncertainty, assumptions, and open questions visible; add tests around review-required wording | Before Phase 27D |
| Documentation consolidation hiding important safety docs | Simplifying docs could bury limitations, privacy, and no-AI boundaries | Keep safety docs linked from README, docs index, and release docs; avoid deleting historical safety docs | Before Phase 27E |
| npm planning becoming npm publish | Planning could be misread as approval to publish a package | Keep `package.json` private and require separate owner approval before publication | Before any npm-related phase |
| AI runtime gate design being mistaken as implemented AI runtime | Design notes could imply provider-backed extraction exists | Label runtime gate work as design-only unless a later phase explicitly approves implementation | Before any AI preview phase |

## Review Cadence

- Review before Phase 26.
- Review before each Phase 27 sub-phase.
- Update if public feedback introduces new safety or usability risks.
- Do not close risks only because tests pass.
