# v0.3 Risk Register

Status: Phase 27E risk register with docs consolidation completion note.

This register covers the proposed v0.3 baseline and stretch areas. It does not start implementation.

Phase 26 planning outcome:

- CI implementation requires separate approval.
- Review/apply polish must not become auto-apply.
- Handoff polish must not imply implementation authorization.
- Workflow inference must preserve uncertainty.

Phase 27A outcome:

- CI implementation was separately approved.
- CI remains validation-only.
- Future CI expansion still requires separate approval.

Phase 27B outcome:

- Review/apply-plan readability improved without auto-apply.
- Manual plan output remains owner-reviewed and final-spec-pack-free.
- `review:quality` covers the local fixture flow without network or AI calls.

Phase 27C outcome:

- Handoff pack readability improved without agent calls, task generation, or implementation authorization.
- Machine artifacts are described as traceability inputs, not generated code.
- `handoff:quality` covers the local fixture flow without network or AI calls.

Phase 27D outcome:

- Workflow inference now separates observed status labels, transition candidates, branch/exception candidates, and open workflow questions.
- Ambiguous reopened/rejected/approved behavior remains review-required and question-driven.
- `workflow:quality` covers the staff and CRM draft flows without network or AI calls.

Phase 27E outcome:

- Documentation navigation now separates first-time, canonical, workflow, AI preview design, release record, v0.3 planning, and historical planning paths.
- Docs Maintenance Guide and Docs Status Map make current-vs-historical status explicit.
- Safety and limitations docs remain linked from README, Start Here, Docs Index, and Docs Status Map.

| Risk | Impact | Mitigation | Review Point |
|---|---|---|---|
| CI scope creep into release automation | CI work could accidentally create tag, release, publish, or deployment behavior | Keep CI focused on validation only; Phase 27A adds no publish, release, tag, deploy, secrets, artifact, or schedule behavior | Before any CI expansion |
| Review UX polish becoming auto-apply behavior | Users may think clearer review/apply-plan output authorizes automatic changes | Keep review outputs human-readable and manual; keep auto-apply out of baseline | Before any future apply-plan expansion |
| Handoff readability being mistaken as implementation authorization | Handoff packs could be treated as instructions to start coding | Preserve context-only language and no-agent/no-code-generation boundaries | Before any future handoff expansion |
| Workflow inference overclaiming uncertain transitions | Heuristics could turn weak signals into false workflow requirements | Keep uncertainty, assumptions, branch candidates, and open questions visible; keep `workflow:quality` in the baseline checks | Before any future workflow inference expansion |
| Documentation consolidation hiding important safety docs | Simplifying docs could bury limitations, privacy, and no-AI boundaries | Keep safety docs linked from README, Start Here, docs index, docs status map, and release docs; avoid deleting historical safety docs | Before any future docs consolidation |
| npm planning becoming npm publish | Planning could be misread as approval to publish a package | Keep `package.json` private and require separate owner approval before publication | Before any npm-related phase |
| AI runtime gate design being mistaken as implemented AI runtime | Design notes could imply provider-backed extraction exists | Label runtime gate work as design-only unless a later phase explicitly approves implementation | Before any AI preview phase |

## Review Cadence

- Review before Phase 26.
- Review before each Phase 27 sub-phase.
- Update if public feedback introduces new safety or usability risks.
- Do not close risks only because tests pass.
