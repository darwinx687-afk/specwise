# Examples Guide v0.1

Status: Phase 11 release-readiness guide.

The examples are synthetic fixtures for local validation and demos. They are not production data and do not contain real personal privacy data.

## Which example should I try?

Start with Legacy CRM Follow-up if you want a clearer business admin scenario.

Use Legacy Staff Evaluation if you want a permission-heavy evaluation workflow.

Use Minimal if you only want to understand the schema shape.

如果你想快速理解业务后台场景，建议先跑 Legacy CRM Follow-up；如果你想看权限和审批更复杂的场景，再看 Legacy Staff Evaluation；如果只想看 schema 结构，可以看 Minimal。

## `examples/minimal`

Purpose: smallest valid spec-pack fixture.

Run:

```bash
node bin/specwise.mjs validate examples/minimal/spec-pack
```

Expected result: validation passes.

Notes: useful for validator sanity checks.

## `examples/legacy-staff-evaluation`

Purpose: realistic legacy staff evaluation scenario with mock input materials and expected output.

Run:

```bash
node bin/specwise.mjs validate examples/legacy-staff-evaluation/expected-output/spec-pack
node bin/specwise.mjs draft examples/legacy-staff-evaluation/input --out ./tmp/draft-test --force
rm -rf ./tmp/draft-test
```

Expected result: expected output validates; deterministic draft is generated with `review_required` status.

Notes: mock screenshots are text files, not OCR or vision inputs.

## `examples/legacy-crm-follow-up`

Purpose: second realistic mock business scenario for a legacy CRM follow-up admin system.

Run:

```bash
node bin/specwise.mjs validate examples/legacy-crm-follow-up/expected-output/spec-pack
node bin/specwise.mjs draft examples/legacy-crm-follow-up/input --out ./tmp/crm-draft --force
node bin/specwise.mjs validate ./tmp/crm-draft/spec-pack
rm -rf ./tmp/crm-draft
```

Expected result: curated expected output validates; deterministic draft is generated with `review_required` status and validates.

Notes: inputs are synthetic mock screenshot text, CSV fixtures, and messy bilingual notes. This example does not use OCR, vision, real AI providers, real company data, or real customer data.

## `examples/invalid`

Purpose: negative fixture proving validation failure behavior.

Run:

```bash
node bin/specwise.mjs validate examples/invalid/missing-buildability/spec-pack --expect-fail
```

Expected result: validation fails as expected because buildability is missing.

## `examples/ai-patches`

Purpose: offline mock AI patch fixtures for patch validation and merge preview.

Run:

```bash
node bin/specwise.mjs patch validate examples/ai-patches/legacy-staff-evaluation.mock-ai-patch.json
```

Expected result: patch validation passes.

Notes: fixture is not generated by a real provider.

## `examples/reviews`

Purpose: human review decision fixture for the legacy staff evaluation merge preview.

Run:

```bash
node bin/specwise.mjs review validate examples/reviews/legacy-staff-evaluation.review-decisions.example.json
```

Expected result: review decisions validation passes.

## `examples/apply-plans`

Purpose: manual apply plan fixture generated from reviewed patch decisions.

Run:

```bash
node bin/specwise.mjs apply-plan validate examples/apply-plans/legacy-staff-evaluation.manual-apply-plan.example.json
```

Expected result: manual apply plan validation passes.

Notes: plan does not auto-apply anything.

## `examples/handoff`

Purpose: example handoff manifest for agent handoff pack skeleton.

Run:

```bash
npm run cli:handoff:manifest:example
```

Expected result: handoff manifest example validation passes.

Notes: example manifest marks implementation as not allowed.

## `examples/config`

Purpose: safe config fixtures for mock provider and future-looking placeholder provider boundaries.

Run:

```bash
node bin/specwise.mjs provider doctor --config examples/config/specwise.mock.config.json
node bin/specwise.mjs provider doctor --config examples/config/specwise.openai-compatible.placeholder.json
```

Expected result: config doctor passes, but placeholder provider remains guarded from extraction.

Notes: configs use placeholder key names and invalid example URLs only. No real API keys should be committed.
