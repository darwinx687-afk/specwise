## 1. Purpose

Select a proposed baseline scope for SpecWise v0.3 without starting implementation.

This document uses the Phase 24 post-v0.2 feedback snapshot, v0.3 planning input, known limitations, risk notes, and read-only GitHub issue checks.

## 5. Recommended v0.3 Baseline Scope

Recommended v0.3 baseline scope:

1. GitHub Actions CI planning and optional CI implementation
2. Review/apply-plan usability polish
3. Handoff pack readability polish
4. Better deterministic workflow inference
5. Documentation consolidation after v0.2

v0.2 improved examples, deterministic drafts, CSV inference, CLI/DX, and first-time docs.
v0.3 should strengthen reliability, review usability, handoff readability, and public repo maintainability before adding high-risk AI/runtime features.

v0.2 已经改进示例、确定性草案、CSV 推断、CLI/DX 和新手文档。v0.3 更适合先增强可靠性、审查体验、handoff 可读性和公开仓库维护能力，而不是过早加入高风险 AI/runtime 功能。

## 6. Stretch Candidates

Stretch candidates:

Stretch items should not enter implementation without a separate Phase 26/27 decision.

- auto-apply patches
- final spec-pack generation
- production-ready claims

## 8. Safety Boundaries

v0.3 scope selection preserves the current SpecWise preview boundaries:

- no real AI provider calls by default
- no prompt runner
- no OCR / vision
- no Web UI
- no coding agent calls
- no generated application code
- no auto-apply AI patch
- no final spec-pack generation
- no npm publish by default
- no new dependency without separate approval
- no production-ready claim

Handoff packs remain context-only. Review/apply-plan polish must not become automatic application behavior.

## 9. Implementation Prerequisites

Before v0.3 implementation starts:

- review current GitHub issues again
- confirm the baseline scope with the repo owner
- decide whether CI is planning-only or implementation
- confirm no new dependencies unless separately approved
- confirm npm publish remains out of scope
- keep `package.json` private unless separately approved
- preserve v0.2 validation scripts
- preserve no-AI / no-agent boundaries
- break work into small phases

## 10. Release Criteria

Any future v0.3 preview release should require:

- selected scope implemented through separate approved phases
- all existing v0.2 validation commands still passing
- new checks for any added workflow polish
- hygiene scan with no local paths, secrets, dependency folders, or temporary artifacts
- docs updated without hiding safety boundaries
- owner approval before tag or release creation
- npm publication still excluded unless separately approved
