# v0.3 Implementation Plan

Status: Phase 27C implementation plan with handoff readability completion note.

This is a planning document.
No v0.3 implementation starts in Phase 26.

这是规划文档。Phase 26 不开始 v0.3 功能开发。

## 1. Purpose

This document turns the Phase 25 v0.3 baseline scope into an implementation planning outline for later Phase 27 work.

Phase 26 defines phase sequencing, workstream details, acceptance criteria, test strategy, safety boundaries, dependency policy, CI decision boundaries, and release criteria.

It does not implement v0.3 behavior.

## 2. v0.3 Baseline Scope

Accepted baseline from Phase 25:

1. GitHub Actions CI planning / optional CI
2. Review/apply-plan usability polish
3. Handoff pack readability polish
4. Deterministic workflow inference polish
5. Documentation consolidation after v0.2

Current public issue feedback: none yet.

The baseline remains based on project maturity, known limitations, and release-readiness gaps rather than external user demand.

## 3. Phase Breakdown

Recommended later phases:

- Phase 27A: GitHub Actions CI / Reliability Planning or Implementation
- Phase 27B: Review and Apply-plan Usability Polish
- Phase 27C: Handoff Pack Readability Polish
- Phase 27D: Deterministic Workflow Inference Polish
- Phase 27E: Post-v0.2 Docs Consolidation
- Phase 27F: v0.3 Release Readiness

Phase 27A CI implementation was owner-approved.
The CI workflow is intentionally limited to local validation commands.

Phase 27B review/apply-plan usability polish was implemented without adding dependencies, provider calls, agent calls, auto-apply behavior, or final spec-pack generation.

Phase 27C handoff pack readability polish was implemented without adding dependencies, provider calls, agent calls, task generation, code generation, auto-apply behavior, or final spec-pack generation.

## 4. Workstream Details

### Workstream 1: GitHub Actions CI / Reliability

Goal:

Make repository validation easier to run consistently after v0.2.

Candidate content:

- CI design for `node --check`
- `npm test`
- `npm run smoke`
- no dependency install beyond npm built-ins
- no secrets
- no publish
- no release workflow

Boundary:

Phase 27A adds `.github/workflows/ci.yml`.
Any CI implementation must avoid npm publish, release automation, secrets, or external services.

Phase 27A completion note:

The CI workflow runs on pull requests and pushes to `main`, uses `permissions: contents: read`, uses only official GitHub actions, and runs local validation commands without installing dependencies, caching `node_modules`, uploading artifacts, publishing packages, creating releases, creating tags, using secrets, calling AI providers, or calling coding agents.

### Workstream 2: Review / Apply-plan Usability Polish

Goal:

Make review decisions, review reports, and manual apply plans easier to understand and act on.

Candidate improvements:

- clearer review decision summaries
- better blocked item grouping
- better follow-up question formatting
- better manual step prioritization and owner hints
- more explicit "not auto-applied" wording
- more readable reviewed handoff plan
- dependency-free review/apply usability smoke check

Not in scope:

- no auto-apply
- no final spec-pack generation
- no business rule confirmation automation

Phase 27B completion note:

Review reports now group safe carry-forward items, business confirmation items, deferred items, rejected items, blocked readiness reasons, and recommended review actions. Manual apply plans now include priority, suggested owner, high-risk hints, semantic step groups, a Before Editing checklist section, and grouped blocked-items markdown. `npm run review:quality` covers the local fixture flow.

### Workstream 3: Handoff Pack Readability Polish

Goal:

Make handoff pack skeletons easier for humans and future agents to inspect without authorizing implementation.

Candidate improvements:

- better root README summary
- better open questions grouping
- better implementation boundary wording
- better agent-specific safety notes
- better machine/source artifact explanation
- dependency-free handoff readability smoke check

Not in scope:

- no agent calls
- no task generation
- no code generation
- no implementation authorization

Phase 27C completion note:

Generated handoff packs now include clearer root README status/navigation, non-negotiable agent boundaries, grouped open questions, clearer implementation boundaries, grouped manual apply plan summaries, `machine/README.md`, improved agent-specific boundary sections, strengthened handoff validation phrase checks, and `npm run handoff:quality`.

### Workstream 4: Deterministic Workflow Inference Polish

Goal:

Improve workflow/state inference without AI and without overclaiming uncertain transitions.

Candidate improvements:

- better arrow-chain parsing
- better status label grouping
- better uncertain transition detection
- better reopened/rejected/approved branch handling
- better workflow open question generation

Safety boundary:

Uncertain transitions must remain questions.
Do not turn ambiguous workflow notes into confirmed facts.

### Workstream 5: Post-v0.2 Docs Consolidation

Goal:

Reduce documentation overwhelm after v0.2 while preserving safety and traceability.

Candidate improvements:

- clearer docs index grouping
- mark older planning docs as historical where appropriate
- keep first-time path visible
- preserve detailed safety docs
- avoid hiding limitations

Do not delete important historical docs.
Do not remove safety boundaries.

## 5. Acceptance Criteria

Detailed criteria are tracked in [v0.3 Acceptance Criteria](v0.3-acceptance-criteria.md).

High-level criteria:

- CI/reliability plan is documented before any workflow is added
- any future workflow avoids npm publish, release automation, secrets, and external services
- review/apply-plan output becomes easier to scan without becoming auto-apply
- handoff packs remain context-only and validate successfully
- deterministic workflow inference preserves uncertainty
- documentation consolidation keeps first-time, limitations, privacy, and safety docs visible
- `node --check`, `npm test`, and `npm run smoke` pass
- no new dependencies are added unless separately approved
- no real AI provider calls or coding agent calls are introduced

## 6. Test Strategy

Detailed strategy is tracked in [v0.3 Test Strategy](v0.3-test-strategy.md).

Required baseline checks:

Focused checks should be added in later phases only when they match the touched workstream.

## 7. Safety Boundaries

v0.3 implementation planning preserves:

- local-first workflow
- review-required output
- evidence-first claims
- no real AI provider calls
- no prompt runner
- no OCR / vision
- no Web UI
- no coding agent calls
- no generated application code
- no auto-apply patch behavior
- no final spec-pack generation
- no npm publish by default
- no new dependency without separate approval
- no production-ready claims

## 8. Non-goals

Not in Phase 26 and not in the v0.3 baseline:

- real AI provider runtime
- OCR / vision
- PDF parser
- Figma import
- database import
- Web UI
- coding agent calls
- prompt runner
- npm publish by default
- auto-apply patches
- final spec-pack generation
- production-ready claims
- release automation

## 9. Dependency Policy

Default policy: no new npm dependencies.

A dependency may only be considered in a later phase if:

- the repo owner separately approves it
- it is required for an accepted v0.3 workstream
- there is no reasonable dependency-free path
- license and maintenance risks are reviewed
- validation and smoke checks remain green
- the decision is documented before implementation

Phase 26 adds no dependencies.

## 10. Release Criteria

Before any future v0.3 release readiness phase:

- Phase 27A-27E scope has either completed or been explicitly deferred
- all required baseline checks pass
- any new focused checks pass
- hygiene scan passes
- docs clearly state preview limitations
- `package.json` remains private unless npm publication is separately approved
- owner approval exists before any tag or GitHub release
- no release automation is added unless explicitly approved

Phase 26 does not create a tag, create a GitHub release, or publish npm.
