## 1. Purpose

This document selects a conservative v0.2 baseline scope from the existing planning draft, feedback triage system, known limitations, release readiness notes, and current SpecWise v0.1 capabilities.

The goal is to prepare a small, reviewable v0.2 direction before implementation planning begins.

## 5. Recommended v0.2 Scope

Recommended v0.2 baseline scope:

1. More realistic example packs
2. Deterministic draft heuristic polish
3. CSV/XLSX field inference polish
4. CLI error message / first-run DX polish
5. Docs simplification for first-time users

Why this baseline:

- it improves the current local-first workflow
- it strengthens deterministic quality before provider integration
- it makes the project easier to evaluate publicly
- it preserves review-required behavior
- it avoids introducing new runtime trust surfaces too early

Stretch candidates:

- Prompt package preview polish
- Review workflow UX polish
- Handoff pack readability polish
- File parser design for Markdown/CSV/XLSX only

## 6. Explicit Non-Goals

Not in v0.2 baseline:

- real AI provider calls

- auto-apply patches
- final spec-pack generation
- PDF parser
- Figma import
- database connection
- auto implementation

## 7. Safety Boundaries

v0.2 planning must preserve:

- local-first default behavior
- no secret or credential reading
- no real provider URL requirement
- review-required outputs
- evidence-first spec fields
- no hidden prompt runner
- no coding agent calls
- no application code generation
- no final spec-pack generation

Any future opt-in AI provider work must be separately approved and designed after the deterministic baseline improves.

## 8. Implementation Prerequisites

Before Phase 21 implementation planning:

- review any new public feedback
- review GitHub issues again
- confirm the baseline scope
- split baseline items into small phases
- decide whether any parser work can stay dependency-free
- define acceptance criteria per item
- keep `npm test` and `npm run smoke` green
- preserve hygiene scan rules
- confirm no new dependencies unless explicitly justified

## 9. Release Criteria

A future v0.2 preview should not be considered until:

- baseline examples validate
- deterministic draft output improvements are documented
- CSV/XLSX inference behavior is covered by examples or fixtures
- CLI error messages are clearer for first-time users
- first-time docs are simplified
- `node --check` passes
- `npm test` passes
- `npm run smoke` passes
- hygiene scan passes
- known limitations remain visible
- package publishing is still explicitly approved or explicitly skipped

## 10. Open Questions Before Development

- Which example domains best demonstrate messy legacy materials?
- Which deterministic draft heuristics are most confusing today?
- What CSV/XLSX field patterns should be supported first?
- Which CLI errors should be rewritten before adding new behavior?
- Can Markdown/CSV/XLSX parser design remain dependency-free?
- What docs page should a first-time user read first?
- What feedback should change this baseline before Phase 21?
