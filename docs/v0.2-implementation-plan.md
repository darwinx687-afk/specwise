# v0.2 Implementation Plan

Status: Phase 21 implementation planning with Phase 22A, 22B, 22C, 22D, and 22E completion notes. Phase 22F has not started.

This is a planning document.

No v0.2 implementation starts in Phase 21.

这是规划文档。Phase 21 不开始 v0.2 功能开发。

## 1. Purpose

This document turns the Phase 20 v0.2 baseline scope into an implementation planning outline. It defines workstreams, acceptance criteria, test strategy, safety boundaries, dependency policy, and release criteria for later Phase 22 work.

The plan preserves the v0.1 preview boundary: local-first, evidence-first, review-required, no real AI provider calls, no coding agent calls, no app code generation, and no final spec-pack generation.

## 2. v0.2 Baseline Scope

Baseline scope:

1. More realistic example packs
2. Deterministic draft heuristic polish
3. CSV/XLSX field inference polish
4. CLI error / first-run DX polish
5. First-time docs simplification

Stretch candidates:

- Prompt package preview polish
- Review workflow UX polish
- Handoff pack readability polish
- Markdown/CSV/XLSX-only file parser design

## 3. Phase Breakdown

Recommended later implementation phases:

- Phase 22A: Example Pack Expansion (completed)
- Phase 22B: Deterministic Draft Heuristic Polish (completed)
- Phase 22C: CSV/XLSX Field Inference Polish (completed)
- Phase 22D: CLI Error and First-run DX Polish (completed)
- Phase 22E: First-time Docs Simplification (completed, documentation simplification only)
- Phase 22F: v0.2 Release Readiness (not started)

Phase 21 does not start any of these phases. It only prepares the plan.

Phase 22E is documentation simplification only. Phase 22F will be v0.2 release readiness and has not started.

## 4. Workstream Details

### Workstream 1: More Realistic Example Packs

Goal:

Add 1-2 additional realistic mock example packs to show SpecWise beyond legacy staff evaluation.

Phase 22A completion note:

The `examples/legacy-crm-follow-up` pack is now available as the v0.2 baseline example expansion. It uses synthetic mock screenshot text, CSV fixtures, messy notes, curated expected output, an evidence map, and a review-required buildability report.

Candidate examples:

1. Legacy CRM / customer follow-up admin system
2. Legacy inventory / stock movement admin system
3. Internal approval workflow system

Recommended v0.2 baseline: build 1 new example pack. Keep a second example as stretch.

Example requirements:

- synthetic data only
- no real personal data
- no real company data
- mock screenshots as text
- CSV sample
- messy notes
- expected-output spec-pack
- evidence map
- buildability report
- validation must pass

### Workstream 2: Deterministic Draft Heuristic Polish

Goal:

Improve deterministic draft quality without using AI.

Phase 22B completion note:

The deterministic draft generator now has small heuristic polish for module/navigation clues, status/workflow clues, role/action/permission clues, open question categorization, uncertain-rule handling, and readable evidence IDs. `scripts/check-draft-quality.mjs` validates generated drafts for the staff and CRM examples while preserving `review_required`, no AI calls, no dependencies, and no final spec-pack generation.

Candidate improvements:

- better module title extraction
- better status/workflow detection
- better role/action parsing
- better open question generation
- better assumption/question separation
- better evidence ID naming
- more consistent bilingual summary generation

Explicitly not in scope:

- no AI calls
- no model scoring
- no OCR
- no vision
- no fuzzy external parsing dependency

### Workstream 3: CSV/XLSX Field Inference Polish

Goal:

Improve field/entity inference from structured files while staying dependency-free.

For v0.2, CSV inference polish is baseline. XLSX support remains design-only unless a dependency-free path is confirmed.

Phase 22C completion note:

CSV draft generation now uses dependency-free field inference for readable entity names, field types, semantic hints, required hints, relationship hints, and observed enum/status values. Generated entity Markdown includes a field table for review. `scripts/check-field-inference.mjs` covers staff evaluation and CRM CSV fixtures. XLSX remains inventory-only and design-only: no XLSX parser, parser dependency, or runtime spreadsheet extraction was added.

Candidate improvements:

- better field type guessing from headers and values
- better entity naming from filenames
- better ID/date/status/role field detection
- better relationship hints from `*_id` fields
- better required/optional field hints

No dependency should be added without a separate decision.

### Workstream 4: CLI Error / First-run DX Polish

Goal:

Make first-time use clearer and failure states easier to understand.

Candidate improvements:

- more actionable validation errors
- better missing file messages
- better `--help` examples
- better command suggestions
- clearer output folder / `--force` errors
- consistent success summary format
- first-run guide command path

Safety boundaries:

- do not hide `review_required`
- do not make commands look like they call AI
- do not imply final spec-pack generation

Phase 22D completion note:

CLI errors now use a consistent `SpecWise error` format with command usage and next-action guidance for unknown commands, missing arguments, missing paths/files, and non-empty output folders. Major success summaries are more consistent while preserving review-required and no-provider/no-agent boundaries. `scripts/check-cli-dx.mjs` is wired into `npm test`, and `docs/first-run-guide-v0.2.md` documents the first-run path.

### Workstream 5: First-time Docs Simplification

Goal:

Make README and docs easier for first-time visitors.

Phase 22E completion note:

First-time docs now include a shorter README Start Here path, `docs/start-here-v0.2.md`, `docs/command-decision-tree-v0.2.md`, clearer example selection, safety cross-links, and v0.1/v0.2 status callouts. This was documentation simplification only: no runtime behavior, parser support, AI provider call, coding agent call, generated application code, final spec-pack generation, dependency, tag, release, or npm publish was added.

Candidate improvements:

- shorter quick start path
- one-page "Try the legacy example" guide
- "What command should I run?" section
- safety boundary callouts
- v0.1 vs v0.2 status clarity

Do not delete detailed docs. Add or improve navigation instead.

## 6. Test Strategy

Test strategy is tracked in `v0.2-test-strategy.md`.

Required checks for future implementation phases:

- `find bin src scripts -name "*.mjs" -print0 | xargs -0 -n1 node --check`
- `npm test`
- `npm run smoke`
- validate minimal example
- validate legacy example
- validate invalid fixture expected-fail
- validate Legacy CRM Follow-up example expected output
- run draft quality checks for staff and CRM deterministic drafts
- run field inference checks for staff and CRM CSV fixtures
- run CLI DX checks for first-run errors and help output
- smoke handoff flow

No network tests, provider tests, agent tests, or npm publish tests should be added for this baseline.

## 9. Dependencies Policy

Default policy: no new npm dependencies.

A dependency may only be considered in a later phase if:

- it is required for a clearly accepted v0.2 scope item
- there is no reasonable dependency-free path
- the license is acceptable
- the package size and maintenance risk are reviewed
- tests and smoke remain green
- the decision is documented before implementation

XLSX support remains design-only unless a dependency-free path is confirmed or a separate dependency decision is approved.

Phase 22C did not add XLSX parsing. Any future XLSX parser remains subject to this dependency policy.

## 10. Release Criteria

v0.2 release readiness should require:

- all baseline implementation phases complete
- v0.2 acceptance criteria reviewed
- test strategy executed
- hygiene scan clean
- known limitations updated
- README and docs index updated
- changelog prepared
- package remains private unless npm publication is separately approved
- no new tag or GitHub release without explicit release approval
- no real AI provider calls, agent calls, or final spec-pack generation

Phase 22F will cover release readiness and has not started in Phase 22E.
